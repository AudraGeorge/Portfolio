Case Study (Global Systems Architecture)

Deloitte

Scaling Systems for a Global Team

40+ People

Across 4 Continents

The Problem

The challenge was to merge a small, co-located US and Canadian shared services team into a newly-formed global operation supporting the entire firm across multiple continents and time zones.

The existing systems—built for a team that could tap each other on the shoulder—were already straining. Regional teams were duplicating work because they couldn't find existing assets. Projects stalled waiting for approvals across time zones. File naming was inconsistent, making search nearly impossible. There were no shared standards, no asynchronous workflows, and no global taxonomy—everything the distributed organization needed to function as a single, efficient unit.

The Solution & My Work

I architected the foundational information systems needed to support the global expansion. This wasn't a single project—it was building four interconnected systems that together enabled colleagues to work efficiently across continents: a global file structure, a shared taxonomy, asynchronous workflows, and documented governance standards.

Global Information Architecture

The first system to build was a global file structure. The existing naming conventions worked fine when everyone sat in the same office and knew which project "Summer_Campaign" referred to. But scale that across regions running simultaneous campaigns? Chaos.

The Challenge

A designer in the UK searching for "Email_Banner" could pull up dozens of results—but which one was actually approved for their region? Which version was current? Was it compliant with local regulations? The file names gave no clue, forcing people to open files one by one or, more often, just recreate the asset from scratch.

I expanded the file naming convention to include regional identifiers, compliance markers, and version control—turning filenames into metadata that told you everything you needed to know before opening the file.

Globally-Aware Taxonomy

File naming solved the "what is this file" problem, but not the "find all files like this" problem. For that, we needed metadata—standardized tags that could be searched, filtered, and reported on.

The Challenge

Different regions had different compliance needs, different languages, different approval processes. A metadata schema that worked globally had to be consistent enough to enable search across regions while flexible enough to capture regional variations. Too rigid and it wouldn't fit local needs. Too loose and it was useless for search.

I established a global metadata schema with required fields (Region, Language, Status) and optional fields for local customization. This meant a search for "approved email assets" would work globally, while UK teams could add their own Legal_Compliance tag without breaking the system for anyone else.

Artifact: Global Metadata Schema

Field Name Field Type Controlled Values / Description
Region Dropdown (Multi-select) Global, UK, EMEA, APAC, LATAM
Language Dropdown EN-US, EN-GB, FR, ES, JP
Asset_Status Dropdown Global standard: WIP, Review, Approved
Legal_Compliance Checkbox Confirms asset meets local legal review requirements.

Asynchronous Workflow System

File structure and metadata solved the "where is it" problem. But work still moved through people—and people worked across 12-hour time zone gaps. Projects were stalling because coordinators in the US couldn't review work until the UK designer's next day, creating 24-hour delays for every feedback cycle.

The Challenge

The team had a project management tool—but no one used it properly. Updates lived in Slack messages, email chains, and quick hallway conversations. When you're global, there are no hallways. The tool existed, but without a documented process for how to use it asynchronously, people defaulted to synchronous communication (which didn't work across time zones) or just waited until the next day.

I didn't build a new tool—I defined the process that transformed the existing project management system from a neglected ticket tracker into the team's central asynchronous hub. Every project got a dedicated task. All updates, feedback, and asset links had to be posted there. No more "I sent you a Slack message"—if it wasn't in the hub, it didn't exist.

Documented Global Standards

Systems only work if people use them consistently. I collaborated with leadership to document these workflows as formal Standard Operating Procedures (SOPs), creating a centralized reference for how global teams should work together.

These weren't theoretical documents—they codified the practical workflows we'd built, with specific rules like "allow 24-hour review cycles for time zone differences" and "project hubs are the source of truth." New team members could onboard without needing to learn through trial and error.

Artifact: Global SOP Snippet

GLOBAL STANDARD OPERATING PROCEDURE

Cross-Team Asset Handoffs

01
Project Hub is the Source of Truth

All status changes, feedback, and links to assets must be logged in the official project hub task. Communication outside the hub is unofficial.

02
Tag for Region & Language

All assets uploaded to the DAM must include correct `Region` and `Language` metadata tags.

03
Allow 24hr Review Cycle

Due to global time zones, all review deadlines must allow for a minimum 24-hour turnaround.

The Payoff

The result wasn't just a scaled team—it was a unified operating system. Regional teams stopped duplicating work because they could finally find existing assets. Projects moved faster because asynchronous workflows eliminated time-zone delays. New hires onboarded in days instead of weeks because the standards were documented. What started as a co-located US team became a true global operation.
